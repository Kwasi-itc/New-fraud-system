# Data Model Service Extraction Design

## Purpose

This document defines the target design for a standalone Go backend now located under `new/backend/data-model-service` that extracts Marble's data model subsystem into an independently deployable service. The new service will own logical data model metadata and real per-tenant physical PostgreSQL schemas, and it will be consumable by another fraud detection system over a network API.

This is a design-first document. It does not prescribe immediate code movement. It defines the service boundary, migration target, and phased extraction plan.

## Goals

- Create a standalone service responsible for data model management.
- Preserve Marble's current per-tenant physical schema behavior.
- Use PostgreSQL for metadata storage and tenant data schemas.
- Package the service so it can run in its own Docker container.
- Expose a stable API that another fraud system can call without sharing process memory.
- Separate portable domain logic from Marble-specific coupling.
- Make future downstream integrations possible without rebuilding the service.

## Non-Goals for V1

- Rebuilding Marble's full fraud platform.
- Extracting scenarios, rules, workflows, screening, analytics, or AI.
- Providing a shared library as the primary integration model.
- Supporting multiple databases beyond PostgreSQL.
- Implementing cross-system destructive dependency tracking for external consumers.

## Recommended Service Shape

The standalone module should be an HTTP REST service with internal package separation:

- `cmd/server`: process bootstrap
- `internal/app`: wiring
- `internal/domain`: core models and business rules
- `internal/service`: orchestration usecases
- `internal/store`: PostgreSQL persistence
- `internal/tenantdb`: tenant schema management and DDL execution
- `internal/httpapi`: handlers, DTOs, routing
- `internal/migrations`: metadata database migrations
- `pkg/client` later if needed: optional Go SDK for consumers

This keeps deployment decoupled while leaving room for a typed client later.

## Why HTTP Service First

An HTTP service is the best default because:

- it fits the "run in its own Docker" requirement;
- it avoids compile-time coupling with the host fraud system;
- it gives the new service ownership of migrations, schema lifecycle, and compatibility;
- it allows the host system to evolve independently;
- it matches Marble's current service-oriented backend structure better than a shared package.

## High-Level Boundary

The new service should own:

- data model tables
- data model fields
- links between tables
- pivots
- table display and configuration options
- navigation option definitions and index orchestration
- metadata validation rules
- tenant schema creation and mutation
- metadata migrations
- tenant schema migrations generated by model changes

The new service should not own in V1:

- scenario/rule/workflow lifecycle
- external dependency conflict checks against another fraud system
- ingestion execution
- fraud decisioning

## Architecture Overview

There are two persistent concerns:

1. **Metadata database**
- stores organizations, tenant schema registry, data model tables, fields, links, pivots, options, schema migration history, index jobs, and audit records

2. **Tenant physical schemas**
- one PostgreSQL schema per tenant
- contains the real object tables defined by the metadata model
- manipulated by the service when tables, fields, and indexes are created, updated, archived, or deleted

The service must keep these two layers consistent through transactional orchestration and compensating safety rules.

## Tenancy Model

The new service should follow Marble's per-tenant schema pattern:

- one metadata schema, for example `core`
- one tenant schema per organization, for example `tenant_<org_short_id>` or `org_<uuid_suffix>`

Recommended choice:

- use a deterministic schema name derived from the tenant UUID, such as `tenant_<32hex>`
- never allow user-provided schema names
- store the derived schema name in a registry table

Rationale:

- avoids collisions
- avoids rename complexity if tenant names change
- keeps DDL generation deterministic

## Data Ownership Model

Each tenant has:

- metadata ownership in shared metadata tables by `tenant_id`
- physical tables in its own PostgreSQL schema

The service is the sole writer of:

- metadata tables describing the data model
- physical DDL changes in tenant schemas

Other systems should treat the service as the authority for schema structure.

## Domain Modules

### 1. Tenant Registry

Responsibilities:

- create tenant records
- derive and persist schema names
- initialize tenant schema
- track tenant state

Core entities:

- `Tenant`
- `TenantSchema`
- `TenantProvisioningState`

### 2. Data Model Catalog

Responsibilities:

- create, update, read, archive, and delete model tables
- create, update, read, archive, and delete fields
- manage field type rules, enum flags, uniqueness flags, and nullable flags
- manage table metadata such as alias, semantic type, caption field, and classification

Core entities:

- `ModelTable`
- `ModelField`
- `TableClassification`
- `FieldConstraintState`

### 3. Relationship Model

Responsibilities:

- define links between tables
- validate link compatibility
- ensure parent uniqueness rules
- validate path consistency

Core entities:

- `ModelLink`

### 4. Pivot Model

Responsibilities:

- define pivots for object traversal and grouping
- validate field-based and path-based pivots

Core entities:

- `Pivot`

### 5. Presentation Config

Responsibilities:

- store per-table displayed fields
- store field order
- later support table-level UI metadata without coupling to a specific frontend

Core entities:

- `TableOptions`

### 6. Tenant Schema Manager

Responsibilities:

- create tenant schemas
- create physical tables
- add columns
- rename or archive columns
- drop tables and columns
- create and delete indexes
- inspect schema state when needed

Core abstractions:

- `SchemaExecutor`
- `DDLPlanner`
- `IndexManager`

## What Not to Bring Into V1

Do not carry these responsibilities into the first standalone service:

- scenario-aware deletion conflict analysis
- workflow-aware deletion conflict analysis
- analytics settings conflict analysis
- test run conflict analysis
- AI-related summarization or rule assistance
- screening-specific FTM downstream behavior beyond the metadata fields themselves

Instead, expose deletion modes:

- `strict_internal_only`
- `force`
- `archive`

If the host fraud system needs external dependency checks, it should do one of:

- call its own dependency resolver before calling this service;
- or later integrate via an extension hook interface.

## Physical Tenant Schema Rules

For each tenant table created in metadata, create a physical table in the tenant schema.

Default required columns should mirror Marble:

- `id` internal primary key or surrogate key
- `object_id`
- `updated_at`

Recommended V1 physical structure:

- internal column `id uuid primary key`
- required business key `object_id text not null`
- required timestamp `updated_at timestamptz not null`
- custom user-defined fields as typed columns

Recommended uniqueness rules:

- unique index on `object_id`
- optional unique indexes on user fields if allowed by type

## Metadata-to-DDL Contract

Every mutating metadata operation should define whether it requires physical DDL.

Examples:

- create table -> create metadata row + create physical table
- create field -> create metadata row + add physical column
- archive field -> mark archived + rename physical column or make inaccessible
- delete field -> delete metadata row + drop physical column
- create unique field -> create metadata flag + create unique index
- add navigation option -> create metadata config + create index

The service should never mutate metadata silently without deciding the corresponding physical effect.

## Transaction Strategy

Because metadata and tenant DDL may run on different connections and PostgreSQL DDL can have transactional caveats, the service should use a controlled orchestration strategy.

Recommended rule:

- when possible, perform metadata and DDL inside one PostgreSQL transaction if both target the same database instance and the DDL is transactional
- otherwise use a change log and explicit status transitions:
  - `pending`
  - `applied`
  - `failed`

Recommended V1 simplification:

- use one PostgreSQL server for both metadata schema and tenant schemas
- use transactional DDL where PostgreSQL supports it
- keep a schema change log anyway for observability and recovery

## Naming and Validation Rules

Carry over and preserve these rules from Marble:

- table names must be lowercase alphanumeric plus underscore and start with a letter
- field names must follow the same pattern
- reserved fields cannot be deleted casually
- unique fields only allowed for compatible types
- enum and unique constraints are mutually constrained
- link parent fields must be unique and type-compatible
- pivots must resolve to string-compatible identity/grouping fields

These rules belong in domain validators, not in HTTP handlers.

## API Response Strategy

Use explicit domain-shaped payloads. Avoid reusing Marble response DTOs directly. Define:

- stable IDs
- resource version timestamps
- `status` on long-running or recoverable operations
- rich validation errors

Recommended deletion semantics:

- dry-run by default is optional, not implicit
- explicit query or body flag such as `?dry_run=true`
- response should include:
  - whether deletion was performed
  - internal conflicts detected
  - archived resources if applicable

## Authentication and Authorization

Recommended V1:

- service-level bearer token or JWT
- tenant-scoped permissions in claims or API gateway layer

The internal services should depend on an authorization interface such as:

- `Authorizer.CanReadTenant`
- `Authorizer.CanWriteTenant`
- `Authorizer.CanAdminTenant`

Do not port Marble's security implementation directly. Recreate the interface shape, not the current concrete dependency graph.

## Recommended Deletion Model for V1

Current Marble deletion logic is deeply coupled to scenarios, workflows, analytics settings, and test runs. That should not be copied into the standalone service.

Recommended V1 delete behavior:

- delete table/field/link/pivot if there are no **internal** metadata conflicts
- internal conflicts include:
  - field participates in link
  - field is reserved
  - link is used by pivot path
  - table still has dependent fields or links where appropriate
- no external fraud-logic awareness

Recommended future hook:

- before destructive mutation, optionally call an `ExternalDependencyChecker`
- host fraud system can implement it later

## Migration Strategy

The new service needs two migration tracks:

1. metadata migrations
2. tenant schema evolution

### Metadata Migrations

Use a standard Go migration tool such as:

- `golang-migrate`
- or `goose`

Recommended:

- SQL-first migrations
- one baseline migration
- additive migrations after baseline

### Tenant Schema Evolution

Tenant schemas evolve through application-driven operations rather than static global migration files. The service should:

- record every tenant DDL change
- support replay or reconciliation
- provide an admin command to audit tenant schema consistency against metadata

Recommended admin capabilities later:

- `reconcile-tenant-schema`
- `verify-tenant-schema`
- `repair-index-jobs`

- Go API server

## Risks and Tradeoffs

### 1. Metadata and DDL consistency

Risk:

- metadata says a field exists but tenant DDL failed

Mitigation:

- transactional DDL when possible
- schema change log
- reconcile command

### 2. Over-coupling to Marble

Risk:

- copying current usecases directly imports unnecessary abstractions

Mitigation:

- redesign interfaces for the new service instead of transplanting `Usecases`, `ExecutorFactory`, and Marble security

### 3. Premature support for external dependency checks

Risk:

- standalone service becomes coupled to a specific fraud engine

Mitigation:

- keep V1 internal-only
- add extension point later

### 4. Tenant schema naming and DDL safety

Risk:

- SQL injection or naming collisions in dynamic DDL

Mitigation:

- deterministic generated schema names
- strict identifier validation
- quoted identifiers only through safe builders

## Decision Summary

The correct standalone extraction target is not "Marble's existing data model usecase copied into a new folder." The correct target is a new service with:

- its own metadata database schema
- its own tenant physical schema manager
- its own API
- domain rules derived from Marble
- minimal coupling to fraud-specific downstream modules

This gives another fraud detection system a reusable schema-management module instead of a partial transplant of Marble internals.
